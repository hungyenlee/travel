# 全站左側側邊欄導覽，行程清單合併為可展開下拉

## 決策

把原本各頁寫死在右上角的頁首導覽（`台灣散步筆記 ＋ 探索城市 ＋ 我的行程`）改為**全站共用的左側側邊欄**，並將「我的行程」與「行程清單」合併成一個**可展開的下拉項目**。

要點：

1. **共用元件、動態渲染**：新增 `assets/js/sidebar.js`，在四頁的 `#app-sidebar` 內動態產生側邊欄，取代四份重複的 `<header>` 標記。行程清單即時讀 `TripStore`。
2. **版面骨架**：四頁 `<body>` 改為 `.app-shell > (aside#app-sidebar + .app-content)`；`.app-content` 內維持原本的 `<main class="container">` 與頁尾。`<body data-page="explore|plan">` 標示當前頁。
3. **「我的行程」＝可展開下拉**：點展開鈕露出所有行程（縮排、當前高亮、**不顯示數字**），底部固定「＋新行程」。行程頁預設展開、其餘頁預設收合。標籤本身仍是連往 `plan.html` 的連結。
4. **建立行程無死角**：`＋新行程` 就地輸入名稱 → `TripStore.createTrip`（會設為 current）→ 若在行程頁就地開啟（透過 `window.PlanPage.selectTrip` hook，免整頁重載），否則導向 `plan.html` 開啟該新行程。空狀態展開仍看得到「還沒有行程 ＋ 新行程」。
5. **手機版漢堡選單**：窄畫面（`max-width: 767px`）側邊欄收合，頂列只留漢堡鍵＋站名；點開由左側滑出抽屜＋半透明遮罩，點遮罩／✕／Esc 關閉。
6. **行程頁管理列簡化**：`plan.js` 移除原本切換行程的 `<select>` 與「新行程」鈕；頁面標題改顯示當前行程名稱，旁邊保留「重新命名／刪除」。切換與建立都改由側邊欄負責；資料變動後以 `window.Sidebar.refresh()` 同步側邊欄清單與高亮。
7. **「加入行程」面板不動**：城市頁／詳細頁的「＋ 加入行程」按鈕與其 popover（含裡面的建立新行程）維持不變，與側邊欄的建立入口互補（一個是「邊逛邊收藏」，一個是「純建立／切換」）。

## 為什麼記錄

這改動了兩條原本的基本決策：導覽原本刻意只留「首頁＋探索城市」（[decisions.md](../decisions.md) #18），而「我的行程」原是頂部導覽的一個連結、行程切換靠頁內 `<select>`（#27）。導覽是全站骨架，日後看程式的人會問「為什麼從頂部導覽改成側邊欄、為什麼行程清單塞在導覽裡」，把取捨記下來。

## 理由與取捨

1. **為什麼改側邊欄**：行程數量變多時，頂部一個 `<select>` 只能看到當前一個、切換要多一步；側邊欄能一眼列出全部行程並一鍵切換，也讓「建立／切換行程」在全站任何頁面都隨手可及。
2. **為什麼把行程清單併進「我的行程」下拉**：使用者反饋「我的行程」與「行程清單」概念重疊，兩個並列反而混淆。合成一個可展開項目後，導覽階層更單純，也符合「一個入口通往一件事」。
3. **為什麼「＋新行程」固定在下拉底部**：確保**空狀態也有建立入口**——即使一個行程都沒有，展開後仍看得到「新行程」，不會出現「沒有行程、也找不到地方新增」的死角。這是這次改動要解掉的核心問題。
4. **為什麼保留「加入行程」面板**：那是「邊逛景點邊順手收藏、可順帶開新行程」的主要情境，和側邊欄「純建立／切換」互補；移除會讓逛頁面時要收藏得先跳去側邊欄，體驗更差。
5. **為什麼用 JS 動態渲染而非四份靜態 HTML**：側邊欄含**會變動的行程清單**且四頁共用，靜態複製會四處不一致、且無法反映 localStorage。沿用專案「HTML 放插槽、JS 填內容」的既有精神（[ADR-0003](./0003-shared-layer-per-page-scripts.md)），不引入新相依（無框架、無 build）。
6. **為什麼提供 `window.PlanPage` hook**：在行程頁內切換／建立行程時，若整頁導向 `plan.html` 會閃一下、也丟失捲動位置。側邊欄偵測到行程頁提供的 hook 就改為頁內就地切換，其他頁沒有 hook 則安全退回導向。這讓側邊欄不需要知道行程頁的內部細節（鬆耦合）。
7. **手機為什麼用漢堡抽屜**：窄畫面容不下常駐側欄，抽屜是行動裝置最常見、學習成本最低的收合方式；半透明遮罩＋Esc／點外面關閉符合一般預期。
