# 篩選選項採固定允許清單，而非從資料自動推導

## 決策

篩選按鈕（行政區、景點分類、料理分類、特色標籤）的選項來自 `data.js` 裡一份**手動維護的允許清單（taxonomy）**，而不是掃描所有地點資料自動推導。地點的分類與標籤必須取自這份清單；當維護者在 `data.js` 手打了清單以外的詞，城市頁最上方會浮出一條**警告橫幅**提示不一致。（招牌品項不受此限，維持自由填寫。）

## 為什麼記錄

對這類靜態資訊網站，「從資料自動長出篩選選項」才是最直覺、最省維護的做法。我們**刻意選了相反的路**：多維護一份清單、還加打字防呆。未來接手的人很可能會想「幹嘛不自動抓就好」而把它改掉，因此需要留下理由。

## 理由與取捨

- 維護者（第一版即作者本人）是用**手動打字**編輯 `data.js`，容易出現「老店／老舖」這類同義或錯字，造成篩選按鈕分裂。固定清單能強制用詞一致。
- 代價是新增用詞時多一道「先登記到清單」的手續，以及需要一段驗證邏輯來產生警告橫幅。作者接受此代價，換取用詞一致性。
- 此決定容易往回走：未來若想改成自動推導，把篩選來源從清單換成掃描資料即可，成本低。記錄的重點在於「這是刻意選擇」，而非不可逆。

## 相關

- 警告橫幅是給**維護者**看的安全網，與**遊客**無關（遊客沒有任何輸入，只點清單產生的按鈕）。見 `CONTEXT.md` 的「角色」與「分類與標籤」。
