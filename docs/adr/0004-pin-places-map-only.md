# 固定地點只影響地圖；列表與摘要維持誠實；篩選不再自動縮放

## 決策

城市頁（`city.js`）新增「**固定地點**」功能：讓遊客把有興趣的地點釘住，切換篩選後仍留在地圖上，方便看「某景點附近有哪些美食」。搭配這個功能，也調整了地圖的自動縮放行為。

具體規則：

1. **固定是一組地點、可跨類型**：`pinnedIds` 以地點 id 當 key 的集合，可同時固定景點與美食、可固定多個。只存記憶體，重新整理／離開頁面即清空（第一版不做持久化，與規劃書一致）。
2. **固定只影響「地圖」**：
   - 地圖畫「**篩選結果 ∪ 已固定**」（去重）。
   - 被固定的圓點保留類型色（藍／橘），另加**紫色粗外環**並略放大，以資辨識（`--color-pin`）。第一版仍用圓點，未來可換成圖片圖示。
3. **列表與摘要維持誠實**：卡片列表與「共收錄 N 個」摘要**仍嚴格等於篩選結果**，不會把被篩掉的固定地點塞進來。若某固定地點**本來就符合**目前篩選，它的卡片會**排到列表最前面**並加**紫色粗框**。
4. **操作**：固定／取消固定的按鈕放在**地圖預覽卡**上（桌機浮動卡與手機底部卡共用，由 `renderPreview(place, pinned)` 產生、`city.js` 接事件）。「清除篩選」只清篩選、不動固定；有固定時另出現「**清除固定（N）**」按鈕。
5. **地圖縮放改為只在首次渲染自動框選一次**（`didInitialFit` 旗標）。之後任何篩選／固定都**不再自動縮放或移動**地圖；使用者要重新全覽時按地圖上的「**顯示全部**」控制項（對「篩選結果 ∪ 已固定」重跑 `fitBounds`）。

## 為什麼記錄

這個功能牽動了兩條原本寫死的行為，未來看程式的人很可能會「順手改回去」，所以記下取捨：

- 原本 `applyFilters()` 用**同一份陣列**同時餵給「摘要／地圖／列表」（見 decisions.md #14）。現在地圖多疊了一層「固定」，跟列表／摘要**刻意不同步**。看到「為什麼地圖上的點比列表多？」時，這是有意為之，不是 bug。
- 原本**每次篩選都自動 `fitBounds`**（decisions.md #11）。現在改成只在首次。若有人覺得「切篩選地圖不動怪怪的」而想加回自動縮放，會直接破壞固定功能的核心價值（見下）。

## 理由與取捨

1. **列表／摘要為什麼不跟著固定走**：如果固定的景點硬留在「美食」列表最上面，會出現「切到美食卻看到一個景點、數字也對不上」的矛盾。讓列表與數字永遠等於篩選結果，語意最乾淨；固定純粹是**地圖上的空間探索輔助**，符合規劃書「地圖看位置、列表用來瀏覽比較」的分工。
2. **篩選為什麼不再自動縮放**：固定的核心情境是「固定景點 A → 切到美食 → 看 A 附近有什麼吃的」。若切篩選時自動 `fitBounds` 只框美食，A 可能被推出畫面，功能就失效了。最單純可靠的解法是：**切篩選時地圖完全不動**，固定的標記就靜靜留在原地理位置，A 自然還在畫面上。代價是使用者手動放大後沒有自動回全覽，因此補一顆「顯示全部」讓使用者自己觸發。
3. **固定圓點為什麼保留類型色再加外環**：既要看得出它是景點還是美食（類型色），又要看得出它被釘住（紫色外環），兩個資訊都保留。統一用紫色當「已固定」的視覺語言（外環、卡片粗框、按鈕、圖例一致）。
