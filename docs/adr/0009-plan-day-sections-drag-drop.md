# 行程規劃層改為「分天區塊」並支援拖曳排序

## 決策

把「我的行程」規劃層從**單天檢視**（一次只顯示一天、靠天數列切換、卡片用 `<select>` 改天）
改為**分天直向清單**：一頁列出所有天（第 1 天…第 N 天，最後是「未分配」），
每天一段、段內卡片可**拖曳排序**（同天內重排、跨天搬移），並加大地圖在版面的占比。

要點：

1. **分天區塊、一頁看全趟**：右側清單依天分段，每段是獨立的拖曳落點；段標題顯示「第 N 天 · M 個地點」、可折疊。取代原本一次只看一天的天數列。
2. **拖曳握把為唯一拖曳起點**：卡片左側六點握把 `⋮⋮` 才能起拖，避免與「點卡片高亮標記」「點標題進詳細頁」衝突。滑鼠移動 >5px、觸控**長按 200ms** 才啟動拖曳（長按前的觸控移動視為頁面捲動）。
3. **原生 Pointer Events、不引外部套件**：用 `pointerdown/move/up` + `setPointerCapture` 自製拖曳，一套邏輯同時涵蓋滑鼠與觸控。延續「外部相依只有 Leaflet」的專案原則（見 [ADR-0003](./0003-shared-layer-per-page-scripts.md)）。
4. **資料層新增 `TripStore.moveItem(tripId, placeId, targetDay, index)`**：同天重排與跨天搬移的唯一寫入路徑。`targetDay` 夾在 `0..days`、`index` 夾在 `0..該天數量`。**資料模型不變**——同一天的相對順序仍＝這些 item 在 `items` 扁平陣列中的先後（見 [ADR-0006](./0006-trip-planning-localstorage.md)）。
5. **地圖仍只畫「選中的那天」**：延續 [decisions.md](../decisions.md) #28 的原則；改由**點段標題**切換選中天，拖曳 drop 後自動把地圖切到落點那天並 auto-fit。卡片↔標記雙向高亮沿用既有邏輯。
6. **`<select>` 改天收進卡片的「⋯」選單**：連同「上移／下移／移除」一起，作為手機、鍵盤與無障礙的等效重排路徑（不強迫一定要拖曳）。
7. **地圖占比加大**：桌機地圖由約 55% 提高到 62%、拉高視野；手機維持地圖在上、清單在下的堆疊。

## 為什麼記錄

原設計（[decisions.md](../decisions.md) #28）刻意用「單天檢視 + 天數列切換」，本次把它換成「分天區塊 + 拖曳」，
是規劃層互動的一次明顯改版，且新增了資料層方法與一套自製拖曳，值得把取捨留給日後的人。

## 理由與取捨

1. **為什麼要先改版面，才做拖曳**：拖曳的直覺是「把卡片從 A 拖到 B」，需要同時看見來源與目標。原本一次只看一天根本看不到目標天，拖曳無從施力。所以先把版面改成「一頁看到整趟」，拖曳才成立。
2. **為什麼選分天直向清單、不做看板橫向多欄（Trello 式）**：地圖是這頁的靈魂，直向版能讓地圖 `sticky` 固定在左側、隨清單捲動保留視野；看板版為了騰出欄寬得把地圖壓扁或收起，反而弱化既有的地圖↔清單雙向連動。直向版也與手機版同構（手機本來就直向），不必維護兩套。看板版列為日後可選的「檢視切換」，非第一版。
3. **為什麼不引拖曳套件（如 SortableJS）**：專案定位純靜態、外部相依只有 Leaflet。原生 Pointer Events 足以涵蓋滑鼠＋觸控，且對觸控「長按才拖、否則捲動」這種細節更可控；多一個相依不划算。
4. **為什麼保留 `<select>`／上下移到「⋯」選單**：純拖曳對鍵盤與部分無障礙情境不友善。把「移到某天／上移／下移」保留為等效操作，是誠實的無障礙路徑，也是手機上的備援。
5. **為什麼新增 `moveItem` 而非沿用 `setPlaceDay`**：`setPlaceDay` 只改 `day`、不動位置，無法表達「同天內排到第幾個」。`moveItem` 用「先移除、改天、再插到該天第 index 個對應的扁平位置」一次覆蓋同天重排與跨天搬移，且不改動既有 localStorage schema。`setPlaceDay` 仍保留給「⋯」選單的「移到某天末端」。
