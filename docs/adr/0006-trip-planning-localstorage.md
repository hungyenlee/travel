# 行程規劃採純前端 localStorage、多個命名行程

## 決策

新增「行程規劃」功能：遊客可建立多個命名行程，把想去的地點分配到「第 N 天」。整個功能是**純前端、無後端**的，所有資料存在瀏覽器 localStorage，資料永不離開這台瀏覽器。

要點：

1. **持久化用 localStorage，不用後端**：所有行程存在單一 key `tsn.trips.v1`（`tsn` = 台灣散步筆記）。重整、關分頁、隔天再開都還在；但**換裝置不同步、不做分享連結**。
2. **多個命名行程**：一個 key 底下存一個 `trips` 陣列，每個行程各有 id、名稱、天數與地點清單。支援「A 存檔 → 規劃 B → 回頭編輯 A」，並記住「上次檢視的行程」（`currentTripId`）。
3. **行程可跨城市**：行程只是一堆地點 id（＋各自的天數安排），不綁城市。第一版只有台南，跨城市實際還用不到但不擋。
4. **抽象天數（第 N 天）＋未分配桶**：不綁真實日期。`day` 值 `0` = 未分配、`1..N` = 第 1..N 天；加天永遠加在最後、減天時該天地點退回未分配並自動連號重排。
5. **孤兒地點靜默略過**：讀取時若地點 id 在 `places` 找不到 → 不顯示、頁面不壞，頂端顯示「已略過 N 個」小提示。

### 資料模型

存在 `localStorage["tsn.trips.v1"]`，值為 JSON：

```json
{
  "version": 1,
  "trips": [
    {
      "id": "t...",
      "name": "行程名稱",
      "createdAt": 1712345678,
      "days": 1,
      "items": [ { "placeId": "chihkan-tower", "day": 0 } ]
    }
  ],
  "currentTripId": "t..."
}
```

- `version`：schema 版本，日後結構若變可據此遷移。
- `trips[].days`：天數 N（`>= 0`）；`0` 代表只有「未分配」。
- `items[].day`：`0` = 未分配、`1..N` = 第 1..N 天。同一行程內一個 `placeId` 最多出現一次。
- `currentTripId`：上次檢視的行程 id，可為 `null`。
- **讀取時若 JSON 損毀或不存在** → 一律視為空狀態 `{version:1,trips:[],currentTripId:null}`，不得丟例外。

## 為什麼記錄

這是第一個「遊客會產生並保存資料」的功能，牽動了幾條專案原本的基本前提（見 [CONTEXT.md](../../CONTEXT.md) 對「遊客」的定義：原本第一版連輸入框都沒有）。未來看程式或想擴充的人，很可能會問「為什麼不乾脆做個後端／登入／分享連結？」，所以把取捨記下來。

## 理由與取捨

1. **為什麼是 localStorage、不是後端**：專案定位是純靜態網站（GitHub Pages、無伺服器、無資料庫、無帳號）。行程規劃是個人的、暫時的筆記，localStorage 剛好夠用，且完美延續「資料不離開瀏覽器、零帳單風險」的既有精神（同 [ADR-0004](./0004-pin-places-map-only.md) 固定地點只存記憶體的思路，只是這裡需要跨工作階段留存，所以升級為 localStorage）。代價是換裝置不同步——這是刻意接受的第一版限制。
2. **為什麼不做分享連結**：把行程分享給同行者需要「網址編碼」或「後端儲存」。網址編碼會讓連結又長又脆、且仍無法協作編輯；後端則違背無伺服器的定位。第一版明確不做，留待日後（見 decisions.md「尚未採用」）。
3. **為什麼用抽象天數而非真實日期**：真實日期需要日曆、出發日、跨月處理等一整套 UI 與邏輯，且多數人排行程時其實是先想「大概分三天、這幾個放第一天」，日期反而是後期才敲定。抽象的「第 N 天」桶用最小的複雜度覆蓋核心需求；真實日期列為未來項目。
4. **孤兒地點如何處理、為什麼**：行程存的是地點 id，而地點資料由維護者手動編輯 `data.js`，可能被改名或刪除。若讀到 id 對不到地點就讓頁面壞掉，體驗最糟。第一版選擇**靜默略過不顯示**，並在頂端給「已略過 N 個」的小提示——這是給維護者的安全網（比照 [ADR-0001](./0001-fixed-taxonomy-with-warning.md) 清單外標籤的警告精神），對一般遊客幾乎無感。這個處理方式日後可能改（例如保留失效卡片、可手動清除），先求頁面不壞。
